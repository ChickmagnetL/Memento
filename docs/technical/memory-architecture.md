# Memento 记忆系统

一个只有检索能力的对话系统，能回答「这条视频里说了什么」，却回答不好「总结一下我存的相关视频」。前者是检索，后者需要记忆。

Memento 是一个把视频内容转化为可检索知识库、再通过对话检索知识的应用。它的对话系统底层不是单一的 RAG 链路，而是一套分层记忆架构——在「记住更多」和「上下文不爆炸」之间取得平衡，在「答得准」和「答得全」之间做出区分。

本文剖析这套记忆系统的设计。

---

## 为什么对话需要一套记忆系统

Memento 最初的对话功能极为简单：一个 AI 智能体（agent），一个检索工具。用户提问，工具去向量库里捞最相关的几段字幕，AI 据此作答。这能跑通「这条视频里 useState 在哪讲」这类问题。

但真实使用很快暴露出三类问题，它们分别对应三种不同的「记忆缺失」：

| 问题 | 表象 | 根因 |
|------|------|------|
| 对话不持久 | 刷新页面、重启应用，历史对话全部消失 | 会话只存在于进程内存，未持久化 |
| 总结性回答缺漏 | 问「总结这个视频讲了什么」，AI 只拼凑出几段碎片 | 检索只召回最相关的若干片段，AI 却默认那就是全部 |
| 上下文每次清零 | 每开一场新对话，都要重新交代「我在学什么、我要什么风格」 | 系统不跨对话保留关于用户本人的信息 |

这三类问题不能用同一个机制解决。对话持久化、知识的全局视野、用户偏好与画像的长期保留——它们的技术栈、存储介质、触发时机各不相同。强行统一为一个「记忆模块」只会带来耦合与脆弱。

因此 Memento 将记忆系统拆为三个职责清晰、彼此独立的组成部分：

```
┌──────────────────────────────────────────────────────────┐
│                  Memento 记忆系统                         │
│                                                          │
│   ┌────────────┐   ┌────────────┐   ┌────────────┐      │
│   │  会话历史   │   │ 知识摘要层  │   │ 个人偏好    │      │
│   │  记对话     │   │  记知识     │   │  记偏好     │      │
│   └────────────┘   └────────────┘   └────────────┘      │
└──────────────────────────────────────────────────────────┘
```

下面逐层展开。

---

## 一、会话历史

这一层做的事很直白：**把每一轮对话（用户问了什么、AI 答了什么）按原样存到本地数据库，不丢、能翻回。** 用户的每场对话可在界面中切换、新建、删除（当前实现为聊天页 header 的会话下拉，而非左侧栏清单）；重新进入应用时，自动回到上次那场对话。

本文真正的重点，是下面两层的架构设计。

---

## 二、知识摘要层

### 2.1 根因：检索的完整性幻觉

这是三层中最关键的一层，针对的是 RAG（检索增强生成）的一个结构性缺陷。

典型 RAG 的工作方式是：用户提问 → 从知识库召回 top-K 个最相关的片段（例如 5 段）→ AI 基于这 5 段生成回答。对于「useState 在第几分钟讲到」这类指向具体片段的问题，它是胜任的。

但对于「总结一下这个视频讲了什么」，它系统性失效。原因不在召回的「准」，而在召回的「全」：一条视频的字幕被切成几十段，检索只返回其中几段，而 **AI 默认这几段就是全部依据**，于是它的总结必然遗漏它从未看到的内容。

更深层的问题是：**AI 甚至不知道知识库里到底有哪些视频。** 它没有全局视野，无从判断召回是否完整。这是一种「完整性幻觉」——模型没读过全文，却以为自己读过了。

### 2.2 解法：给知识加摘要层，给 AI 全局视野

知识摘要层从两个方向同时入手。

**方向一：为每个文档建立三层结构。**

每个视频在知识库里不只有切碎的字幕片段，而是按抽象程度分为三层（借鉴分层检索的思路，L 表示 Layer，越浅越概括）：

| 层 | 内容 | 角色 |
|----|------|------|
| **L1 原文** | 清洗后的字幕全文（切块） | 细节问题的依据 |
| **L2 摘要** | 该视频的段落摘要（约 150–300 字） | 锁定文档后的全局概括 |
| **L3 概括** | 一句话讲清这视频是什么（≤60 字） | 全局检索的最小单元 |

L1 是既有就有的，L2/L3 是知识摘要层新增的。摘要与概括在**字幕清洗阶段一次生成**——清洗本身就需要读全文，让同一次模型调用顺手产出 L2/L3，边际成本接近零，且保证新视频入库即具备完整三层，不存在「刚导入的视频还没摘要」的空窗。

实现上主路径是 clean 时写入；`summarize_document` 另有 `get_or_generate` 兜底：旧库缺摘要时会现生成并回写（工具表「无则生成」即指此）。设计上仍以入库即生成为准，兜底只服务历史数据。

**方向二：给 AI 三个工具，由它自主决定如何调用。**

知识摘要层不改变既有检索，而是给 AI 额外注册两个工具（加上原有的共三个），由模型根据问题性质自主选择调用哪个：

| 工具 | 作用 | 触发场景 |
|------|------|----------|
| `search_knowledge` | 检索 L1 原文细节片段 | 指向具体片段的问题 |
| `lookup_documents` | 内部对 L3 概括做检索，返回最相关的若干视频清单 | 总结性、探索性问题——先看有哪些视频 |
| `summarize_document` | 返回某视频的 L2 段落摘要（有则直接读，无则生成） | 锁定某视频后深入 |

下面是一次总结性提问的处理流程，左侧是知识摘要层的工作方式，右侧对照仅使用 RAG 检索时的失效路径：

```
        【知识摘要层】                          【仅 RAG 检索（失效）】

用户问「总结一下这个视频」                   用户问「总结一下这个视频」
   │                                            │
   ▼                                            ▼
调用 lookup_documents                          直接 search_knowledge
→ 拿到相关视频的 L3 概括清单                   → 召回 top-K 个 L1 原文片段
→ 知道有哪些视频、各自讲什么                   （只看到几段碎片）
   │                                            │
   ▼                                            ▼
选中目标 → 调用 summarize_document              AI 把这几段当成全部依据
→ 拿到该视频的 L2 全文摘要                      → 直接拼凑总结
   │                                            │
   ▼                                            ▼
基于摘要作答（有全局视野）                     总结必然遗漏从未看到的片段
```

### 2.3 几个关键的设计不变量

**摘要不进入原文检索池。** 摘要的向量存放在独立的索引集合中，与原文片段的检索完全隔离。这保证了一个重要的不变量：**搜细节时，召回的永远是原文片段，不会被摘要「抢」走位置。** 两条检索路径互不干扰。

**`lookup_documents` 内部做检索筛选，而非让 AI 遍历全部 L3 概括。** 一个自然的设计是让 AI 把所有视频的概括都看一遍再挑——这在视频数量少时可行，但视频数量级是不确定的。一旦积累到数百条，全量塞入上下文既会撑爆预算，也会分散模型注意力。因此 `lookup_documents` 在工具内部用向量检索对 L3 概括做筛选，只返回与当前问题最相关的若干条。这条约束直接规避了「全量遍历」的失败模式。

**清洗时一次产出三层，而非按需懒生成。** 懒生成（被问到时才生成摘要）能省下不被访问文档的成本，但会引入两个问题：一是首次总结回答会有明显延迟，二是未生成摘要的新视频在 `lookup_documents` 检索时会被漏掉，破坏全局视野的完整性。考虑到清洗本就读全文、摘要生成的边际成本极低，入库即生成是更稳妥的选择。

### 2.4 与 RAPTOR 的关系

知识摘要层在思想上借鉴了分层检索（RAPTOR 一类方法「用摘要层改善总结性回答」的核心洞见），但刻意裁剪了 RAPTOR 的多层聚类：RAPTOR 的递归聚类是为「跨数千文档构建主题树」设计的，而 Memento 是单用户应用，单文档内的「原文 + 摘要 + 概括」两层抽象已足以覆盖「细节问题」与「总结问题」两种模式。跨视频的主题聚类是一个更重的、可后置的独立问题。

## 三、个人偏好

### 3.1 问题与定位

即便对话能持久化（第一层）、即便知识有了全局视野（第二层），每开一场新对话，AI 仍然不认识你——你得重新交代「我在学 React」「回答给我简洁点」「这块我不熟，多展开」。这是第三类记忆缺失：关于用户本人的、跨对话稳定的信息没有被保留。

需要明确的是，Memento 是一个**知识问答与学习**应用，不是代码生成工具。这意味着它要记的「用户偏好」形态与通用编程助手不同：

| | 编程类助手 | Memento |
|--|-----------|---------|
| 记什么 | 技术栈、项目结构、编码规范 | 学习进度、薄弱点、解释风格偏好 |
| 典型条目 | 「项目用 React」 | 「在学 React」「Hooks 不熟，讲时多展开」 |

Memento 的个人偏好层，本质上是一份**学习画像**。

### 3.2 记什么与不记什么

准入标准是一条简单的判据：**这条信息在未来的新对话里，是否仍然需要 AI 知道？**

- **记**：解释风格偏好、正在学习什么、薄弱环节、长期技术背景。
- **不记**：一次性事实（归会话历史）、易变信息（会过时成噪声）、通用常识（模型本就掌握）、视频内容本身（归知识库）。

这条边界对应「用户主动确认的、跨对话稳定的偏好」，而非「自动从对话中扒取的、会随时间过时的记忆」。后者正是业内多个自动记忆系统最被诟病的缺陷——记忆库随时间积累成一堆过时、矛盾、不可信的条目。Memento 用严格的准入从源头规避它。

### 3.3 写入：双路径

个人偏好有两条并行的写入路径。

**路径一：AI 主动提议，用户拍板。**

1. AI 在每轮回答中，自主判断本轮是否暴露了值得长期保留的偏好信息；
2. 若有，它发起一条提议（携带建议内容），系统标记为「待确认」，**不立即写入、不阻塞当轮回答**；
3. 本轮回答结束后，界面在消息流中展示确认卡片（接受 / 编辑 / 拒绝）；
4. 用户选择**接受、拒绝或纠正**，只有接受或纠正后的内容才真正落库。

**路径二：用户主动声明。**

用户以 `/remember <内容>` 直接写入，无需 AI 提议、无需二次确认——用户主动说出的话，默认即为确认。（记忆面板当前提供列表与删除，不提供「新增」按钮。）

两条路径共同保证了一个核心不变量：**写入偏好的每一条，都经过用户本人的确认。** 系统不会静默地从对话中自动抽取并写入——这是它与「自动记忆」类系统的根本分歧。

### 3.4 注入：每次现拼，偏好库为唯一真相源

偏好条目数量有限（数十条量级），无需向量检索。系统在**每次组织 AI 指令时，从偏好库实时读出全部条目，拼入指令的一个标记块**：

```
（固定指令：你是 Memento ……引用规则……）

<user_memory>
- 正在学习 React，回答相关问题时假设有前端基础
- 偏好简洁，少废话多代码
- useEffect 不熟，讲到时多展开
</user_memory>
```

这里的关键设计是：**偏好库（数据库）是唯一真相源，指令里的偏好块是每次临时拼装的，不是持久存在某处的文件。** 增删改一条偏好，等价于改数据库的一行；下一次组织指令时，拼出来的块自动反映最新状态。代码无需「定位指令里的偏好块、再去改它」——因为那个块根本不是持久化的。

这一选择避开了「在持久文件里维护带标记的记忆段」的复杂度，也与 Memento「指令由代码组装」的现有架构一致。

## 三层的协作

三个组成部分相互独立，却共同支撑一次完整的对话：

```
新对话开始
   │
   ├─ 个人偏好：把用户的学习画像拼入 AI 指令（AI 知道「你是谁、怎么帮你」）
   └─ 会话历史：恢复用户选定的历史会话（或开启新会话）
   │
   ▼
用户提问
   │
   ▼
知识摘要层：
   细节性问题 ──→ search_knowledge（搜 L1 原文片段）
   总结性问题 ──→ lookup_documents（看 L3 全局）─→ summarize_document（读 L2 摘要）
   │
   ▼
AI 作答（带着用户的偏好风格 + 知识的全局视野）
   │
   ├─ 会话历史：本轮对话落库（刷新不丢）
   └─ 个人偏好：若本轮暴露了偏好 ──→ AI 提议 ──→ 用户确认 ──→ 写入（下场对话生效）
```

每一层解决一种独立的记忆缺失。叠加起来，Memento 的对话系统同时具备三种能力：**对话不丢、答得全、还认识你。**

## 附：常见疑问

**这套记忆系统会拖慢回答、占用很多资源吗？**
不会。会话历史存在数据库里，不占 AI 上下文，按需加载；知识摘要存在独立的索引集合里，检索时才召回；个人偏好数十条全量拼入指令，总占用不过几百 token。

**数据安全吗？**
对话、知识库、个人偏好等业务数据保存在用户本机（SQLite / Qdrant / 本地文件）。模型调用仍按 Settings 配置访问 chat / embedding / ASR 端点（可为本地或第三方），与「记忆与知识库是否落本地」是两件事。

**为什么分成三层，而不做成一个统一的记忆系统？**
因为三类问题对应三种本质不同的记忆缺失，技术栈、存储介质、触发时机各异。对话持久化、知识摘要、用户偏好强行合一，只会带来耦合与脆弱。分层让每个部分都简单清晰、可独立演进。
